## 3. Предположения о свойствах программ-нагрузчиков

### 3.1. IO Loader (ioloader)

**Предположения для последовательного доступа (sequence):**
1. Время выполнения должно линейно зависеть от количества повторений
2. Производительность должна быть высокой благодаря эффективному использованию кэша ОС
3. Пропускная способность должна быть стабильной (~300-400 MB/сек) при увеличении количества повторений
4. IOPS (операций в секунду) должны быть относительно постоянными (~90,000-100,000 операций/сек)
5. System time должен составлять небольшую долю от общего времени выполнения (~5-10%)

**Предположения для случайного доступа (random):**
1. Время выполнения должно линейно зависеть от количества повторений
2. Производительность должна быть значительно ниже, чем при последовательном доступе (в 5-10 раз)
3. Пропускная способность должна быть низкой (~40-60 MB/сек) из-за отсутствия эффективного кэширования
4. IOPS должны быть значительно ниже (~10,000-15,000 операций/сек) из-за большого количества операций lseek
5. System time должен составлять большую долю от общего времени выполнения (~80-95%) из-за системных вызовов

**Общие предположения:**
1. При использовании O_DIRECT производительность может снизиться из-за обхода кэша ОС
2. Разница в производительности между последовательным и случайным доступом должна оставаться постоянной при увеличении объема данных
3. Случайный доступ создает значительно больше системных вызовов, что увеличивает system time

### 3.2. CPU Matrix Multiplication (cpu-mat-mul)

**Предположения:**
1. Время выполнения должно линейно зависеть от количества повторений
2. Сложность алгоритма O(n³), где n - размер матрицы
3. При увеличении количества повторений время на одно повторение должно оставаться примерно постоянным
4. Производительность должна быть высокой, так как операция хорошо оптимизируется процессором
5. При компиляции с оптимизацией (-O3) производительность должна значительно возрасти

### 3.3. EMA Replace Int (ema-replace-int)

**Предположения для последовательного доступа (sequential):**
1. Время выполнения должно линейно зависеть от количества повторений
2. Производительность должна быть очень высокой благодаря чтению всего файла в память
3. Пропускная способность должна быть значительно выше, чем при случайном доступе (в тысячи раз)
4. IOPS должны быть очень высокими благодаря эффективной обработке в памяти
5. System time должен составлять значительную долю от общего времени из-за операций чтения/записи больших объемов данных

**Предположения для случайного доступа (random):**
1. Время выполнения должно линейно зависеть от количества повторений
2. Производительность должна быть значительно ниже, чем при последовательном доступе, из-за случайных обращений к диску
3. Количество найденных совпадений зависит от распределения значений в файле и вероятности попадания в нужные позиции
4. Пропускная способность должна быть низкой (~3-4 MB/сек) из-за случайного доступа
5. IOPS будут низкими из-за необходимости выполнения операций lseek, read и write для каждой позиции
6. System time должен составлять большую долю от общего времени из-за большого количества системных вызовов

**Общие предположения:**
1. Последовательный доступ должен быть значительно эффективнее случайного
2. При последовательном доступе обрабатывается весь файл или указанное количество элементов
3. При случайном доступе проверяется только указанное количество случайных позиций

## 8. Многопоточная реализация нагрузчика

### 8.1. Реализация многопоточной версии

Была создана многопоточная версия CPU нагрузчика (`cpu-mat-mul-threaded.c`) с использованием библиотеки `pthread`. Матрица разбивается на строки, каждая из которых обрабатывается отдельным потоком.

**Особенности реализации:**
- Использование `pthread_create` и `pthread_join` для управления потоками
- Распределение строк матрицы между потоками равномерно
- Параметр `--threads` для указания количества потоков

### 8.2. Сравнение однопоточного и многопоточного вариантов

**Параметры тестов:** Матрицы 1000x1000 (для получения времени выполнения не менее 1 минуты)

## 9. Влияние оптимизации компиляции (-O3)

### 9.1. Тесты с оптимизацией

**Параметры тестов:** Матрицы 1000x1000

| Вариант компиляции | Real time | User time | System time | CPU% |
|-------------------|-----------|-----------|-------------|------|
| Без оптимизации (-O0) | ~4.75 сек | ~4.60 сек | ~0.10 сек | 98% |
| С оптимизацией (-O3) | ~2.84 сек | ~2.60 сек | ~0.10 сек | 94% |

**Изменения показателей:**
- **Real time:** сократился на **40.2%** (с ~4.75 сек до ~2.84 сек)
- **User time:** сократился на **43.5%** (с ~4.60 сек до ~2.60 сек)
- **System time:** остался практически неизменным (~0.01 сек)
- **CPU%:** немного снизился (с 98% до 94%), что указывает на более эффективное использование процессора

### 9.2. Объяснение улучшений

**Причины ускорения:**
1. **Векторизация:** Компилятор применяет SIMD-инструкции для параллельной обработки нескольких элементов
2. **Развертывание циклов:** Уменьшение количества проверок условий в циклах
3. **Оптимизация доступа к памяти:** Более эффективное использование кэша процессора
4. **Удаление избыточных вычислений:** Оптимизация общих подвыражений
5. **Инлайнинг функций:** Устранение накладных расходов на вызовы функций

**Выводы:**
- Оптимизация компиляции дает значительное ускорение (более чем в 1.6 раза)
- Улучшения касаются в основном user time, что указывает на оптимизацию вычислений
- System time практически не изменяется, так как оптимизация не влияет на системные вызовы

1. **ioloader** - для нагрузки подсистемы ввода-вывода
2. **cpu-mat-mul** - для нагрузки вычислительной подсистемы
3. **ema-replace-int** - для нагрузки подсистемы ввода-вывода с поиском и заменой

1. **Производительность IO - сравнение типов доступа:**
   - Последовательный доступ к диску **в 8.3 раза эффективнее** случайного доступа
   - При последовательном доступе: пропускная способность ~390 MB/сек, IOPS ~95,000 операций/сек
   - При случайном доступе: пропускная способность ~47 MB/сек, IOPS ~11,765 операций/сек
   - Кэширование ОС существенно улучшает производительность при последовательном доступе
   - При случайном доступе 93-95% времени уходит на системные вызовы (system time)
   - System time при случайном доступе в 16 раз выше, чем при последовательном доступе
   - Разница в производительности остается постоянной при увеличении объема данных

2. **Производительность CPU:**
   - Алгоритм перемножения матриц эффективно использует процессор
   - Производительность стабильна и предсказуема
   - Время выполнения линейно зависит от объема вычислений

3. **Многопроцессные тесты:**
   - Оптимальное количество процессов для системы с 8 ядрами - **4 процесса**
   - При 4 процессах достигается эффективное использование 4 ядер без значительных накладных расходов
   - USER% остается высоким (~86-96%) при любом количестве процессов
   - SYS% увеличивается с количеством процессов, но остается низким (~0-4%)
   - При увеличении количества процессов свыше оптимального real time начинает расти из-за конкуренции за ресурсы

4. **Многопоточная реализация:**
   - Многопоточная версия обеспечивает ускорение в **3.68 раза** при использовании 8 потоков
   - Многопоточная версия эффективнее многопроцессной для одного экземпляра программы
   - Оптимальная конфигурация: одна многопоточная программа с количеством потоков, равным количеству ядер

5. **Влияние оптимизации компиляции:**
   - Компиляция с `-O3` дает ускорение в **1.67 раза** (сокращение real time на 40.2%)
   - Оптимизация в основном улучшает user time (сокращение на 43.5%)
   - System time практически не изменяется при оптимизации
